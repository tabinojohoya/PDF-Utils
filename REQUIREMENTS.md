# PDF-Utils 要件定義書

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| アプリ名 | PDF-Utils |
| プラットフォーム | macOS Tahoe (26.x) |
| 開発言語 | Swift 5 (Swift 6 Concurrency対応) |
| UIフレームワーク | SwiftUI |
| 開発環境 | Xcode 26.2 |
| バンドルID | com.example.PDF-Utils |
| 最小動作OS | macOS 26.2 |

## 2. アプリの目的

複数のPDFファイルを選択し、指定した順序で1つのPDFファイルに結合する、または1つのPDFファイルを複数のファイルに分割するmacOSネイティブアプリケーション。PDF のページ構造を自由に操作する「結合」と「分割」を両輪とした PDF ユーティリティ。

## 3. 機能要件

### 3.1 PDF ファイル追加 (FR-001)

| 項目 | 内容 |
|------|------|
| 概要 | ユーザーが結合対象のPDFファイルを追加できる |
| 入力方法 | ファイル選択ダイアログ（複数選択対応） |
| 入力方法 | ドラッグ&ドロップ（複数ファイル対応） |
| 対象形式 | `.pdf` ファイルのみ |
| バリデーション | PDF形式でないファイルは拒否し、エラーメッセージを表示 |

### 3.2 PDF ファイル一覧表示 (FR-002)

| 項目 | 内容 |
|------|------|
| 概要 | 追加されたPDFファイルをリスト形式で表示 |
| 表示項目 | ファイル名、ページ数、ファイルサイズ |
| サムネイル | 各PDFの先頭ページのサムネイルを表示 |

### 3.3 並び替え (FR-003)

| 項目 | 内容 |
|------|------|
| 概要 | 結合順序をユーザーが自由に変更できる |
| 操作方法 | ドラッグ&ドロップによるリスト内並び替え |
| 操作方法 | 「上へ移動」「下へ移動」ボタン |

### 3.4 ファイル削除 (FR-004)

| 項目 | 内容 |
|------|------|
| 概要 | リストから個別のPDFファイルを除外できる |
| 操作方法 | 選択してDeleteキーまたは削除ボタン |
| 一括操作 | 「すべてクリア」ボタンで全件削除 |

### 3.5 PDF プレビュー (FR-005)

| 項目 | 内容 |
|------|------|
| 概要 | 選択したPDFの内容をプレビュー表示 |
| 表示方法 | 右側ペインにページプレビューを表示 |
| ページ送り | 複数ページのPDFはページ送りが可能 |

### 3.6 PDF 結合実行 (FR-006)

| 項目 | 内容 |
|------|------|
| 概要 | リスト内の全PDFを上から順に結合し、1つのPDFとして出力 |
| 保存方法 | NSSavePanelによる保存先選択ダイアログ |
| デフォルト名 | `Merged.pdf` |
| 処理表示 | 結合中はプログレスバーを表示 |
| 完了通知 | 結合完了後に成功メッセージを表示 |
| エラー処理 | 失敗時はエラー内容を表示 |

### 3.7 結合結果プレビュー (FR-007)

| 項目 | 内容 |
|------|------|
| 概要 | 結合完了後、結果のPDFをQuick Lookまたはプレビューで確認できる |
| 操作 | 「結合結果を開く」ボタンで結合済みPDFを開く |

### 3.8 モード切替 (FR-008)

| 項目 | 内容 |
|------|------|
| 概要 | アプリの動作モードを「結合」と「分割」で切り替えられる |
| 操作方法 | ツールバーのセグメントコントロール（`Picker` with `.segmented`） |
| デフォルト | 結合モード |
| 切替時の挙動 | モード切替時にファイルリスト・選択状態はリセットされる |
| 視覚フィードバック | 選択中のモードがハイライトされ、UIが対応レイアウトに遷移 |

### 3.9 分割対象PDFの選択 (FR-009)

| 項目 | 内容 |
|------|------|
| 概要 | 分割モードでは1つのPDFファイルを入力として選択する |
| 入力方法 | ファイル選択ダイアログ（単一選択） |
| 入力方法 | ドラッグ&ドロップ（複数ドロップ時は先頭1ファイルのみ採用） |
| 対象形式 | `.pdf` ファイルのみ |
| ファイル差し替え | 別のPDFをドロップまたは選択すると現在のファイルを置換 |
| バリデーション | 結合モードと同一（非PDF拒否、破損検出、パスワード保護検出） |

### 3.10 分割方法の選択 (FR-010)

| 項目 | 内容 |
|------|------|
| 概要 | PDFの分割方法を複数のプリセットから選択できる |

| 1ページずつ | 全ページを個別のPDFファイルとして書き出し |
| 偶数/奇数分割 | 偶数ページ・奇数ページをそれぞれ別ファイルに分離 |
| デフォルト | ページ範囲指定 |

### 3.11 分割プレビュー (FR-011)

| 分割境界線 | 分割位置に視覚的な区切り線を表示し、各チャンクを色分け |
| チャンク情報 | 各分割グループのページ数を表示 |
| リアルタイム更新 | 分割設定の変更に応じてプレビューが即座に更新される |

### 3.12 分割実行 (FR-012)

| 項目 | 内容 |
|------|------|
| 概要 | 設定に基づいてPDFを分割し、複数のPDFファイルとして出力 |

| 完了通知 | 分割完了後に成功メッセージを表示（出力ファイル数を含む） |

### 3.13 分割結果の確認 (FR-013)

| 項目 | 内容 |
|------|------|
| 概要 | 分割完了後、出力先フォルダを確認できる |

### 3.14 ページ範囲入力 (FR-014)

| 項目 | 内容 |
|------|------|
| 概要 | ページ範囲指定モードで、分割位置を柔軟に入力できる |
| テキスト入力 | カンマ区切りで範囲を指定（例: `1-3, 4-10, 11-15`） |
| バリデーション | 範囲の重複・ページ番号超過・不正フォーマットを検出しエラー表示 |
| 自動補完 | 未指定ページがある場合、残りのページを自動的に末尾グループに追加 |
| ページ総数表示 | 入力フィールド横にPDFの総ページ数を表示（「/ 24ページ」） |

## 4. 非機能要件

### 4.1 パフォーマンス (NFR-001)

- 100ページ以上のPDFでも数秒以内に結合・分割処理を完了すること
- PDF結合・分割処理はバックグラウンドスレッドで実行し、UIをブロックしないこと
- メモリ効率を考慮し、大容量PDFでもメモリ不足にならない設計とすること

### 4.2 セキュリティ (NFR-002)

- App Sandbox有効（macOS標準のセキュリティモデルに準拠）
- ユーザーが明示的に選択したファイルのみアクセス（`ENABLE_USER_SELECTED_FILES`）
- Hardened Runtime有効

### 4.3 ユーザビリティ (NFR-003)

- macOSネイティブのルック&フィールに準拠
- キーボードショートカット対応（⌘O: ファイル追加、⌘S: 結合実行、⌘⌫: 選択削除）
- ダークモード / ライトモード両対応

### 4.4 信頼性 (NFR-004)

- 破損したPDFファイルについて適切なエラーハンドリングを行うこと
- パスワード付きPDFはスキップし、ユーザーに通知すること

### 5.2 使用フレームワーク

| フレームワーク | 用途 |
|---------------|------|
| SwiftUI | UI構築 |
| PDFKit | PDF読み込み・結合・分割・プレビュー |

| Combine / async-await | 非同期処理 |

> **注**: SwiftDataは本アプリでは不要（永続化不要のため）。テンプレートのSwiftData関連コードは削除する。

### 5.3 PDF結合ロジック

```swift
// 概要: PDFKit の PDFDocument を使用
// 1. 各PDFファイルを PDFDocument として読み込み
// 2. 新規 PDFDocument を作成
// 3. 各ソースの全ページを順番に insert
// 4. 結合後の PDFDocument を write で保存
```

### 5.4 PDF分割ロジック

```swift
// 概要: PDFKit の PDFDocument を使用
// 1. 元PDFを PDFDocument(url:) で読み込み
// 2. 分割設定（SplitConfig）に基づきページ範囲グループを算出
// 3. 各グループごとに新規 PDFDocument を作成
// 4. 対象ページを page(at:) で取得し insert(_:at:) で追加
// 5. 各 PDFDocument を write(to:) で個別ファイルとして保存
// 6. グループ完了ごとに進捗を更新
```

### 5.5 分割設定モデル

```swift
/// 分割方式を表す列挙型
enum SplitMethod: String, CaseIterable, Identifiable {
    case pageRanges   // ページ範囲指定
    case everyPage    // 1ページずつ
    case oddEven      // 偶数/奇数

    var id: String { rawValue }
}

/// 分割設定
struct SplitConfig {
    var method: SplitMethod = .pageRanges
    var rangeText: String = ""  // "1-3, 4-10, 11-15"

    /// 設定からページ範囲グループを算出
    func computeGroups(totalPages: Int) -> Result<[[Int]], SplitConfigError>
}
```

### 5.6 状態管理

- `@Observable` クラス（`PDFMergeViewModel`）で一元管理
- `appMode: AppMode`（`.merge` / `.split`）でモード状態を保持
- 分割モード時: `splitSourceItem: PDFItem?`, `splitConfig: SplitConfig` を追加管理
- SwiftData は使用しない（セッション内の一時データのみ）

```swift
/// アプリの動作モード
enum AppMode: String, CaseIterable, Identifiable {
    case merge  // 結合モード
    case split  // 分割モード

    var id: String { rawValue }
}
```

## 6. GUI設計思想

> *"Design is not just what it looks like and feels like. Design is how it works."*
> — Steve Jobs

本アプリのGUIは「**引き算の美学**」に基づく。機能を足すのではなく、ユーザーが考えなくて済むものを一つずつ取り除く。PDFを結合する——たったそれだけのことに、マニュアルは要らない。アプリを初めて開いた瞬間に、次に何をすべきかが分かる。それが正しいデザインである。

### 6.1 デザイン原則

| # | 原則 | 具体的な指針 |
|---|------|-------------|
| 1 | **極限のシンプルさ** | 画面上の全要素に存在理由を問う。説明できないものは削除する |
| 2 | **一目で伝わる階層** | 最も重要なアクション（結合実行）が視覚的に最も目立つこと |
| 3 | **直接操作の快感** | ドラッグ&ドロップは「掴んで、置く」。人間が物を扱う自然な動作そのもの |
| 4 | **待たせない、退屈させない** | 全てのインタラクションに即座のフィードバック。沈黙は最悪のUI |
| 5 | **一貫性という信頼** | macOSネイティブの作法に従う。独自UIは作らない。Appleが正しい |
| 6 | **余白は贅沢ではなく必然** | 詰め込まない。呼吸できる空間がコンテンツを引き立てる |

### 6.2 カラーシステム

独自カラーは定義しない。macOSのセマンティックカラーに100%準拠する。

| 用途 | カラー指定 | 理由 |
|------|-----------|------|
| 背景 | `.background` (System) | OSのモード切替に自動追従 |
| テキスト（主） | `.primary` | 可読性最優先 |
| テキスト（副） | `.secondary` | 情報の階層を自然に伝える |
| アクセント | `.accentColor` (System Blue) | OSの設定をユーザーが変更可能 |
| 結合ボタン | `.tint` + `.borderedProminent` | 画面内で唯一の「主張する色」 |
| ドロップゾーン（アクティブ） | `.blue.opacity(0.08)` + 破線ボーダー | 受け入れ可能状態を穏やかに示す |
| エラー | `.red` (System) | 赤は「注意」の万国共通語 |
| 成功 | `.green` (System) | 完了の安心感 |
| サムネイル背景 | `.quaternary` | コンテンツを邪魔しない最小限の境界 |

**ダークモード**: 全てのセマンティックカラーがOSレベルで自動反転する。カスタム対応コードはゼロ。

システムフォント（San Francisco）のみ使用。カスタムフォントは不要。

**レイアウトの鉄則**:
- ツールバーの左側は「素材を扱う操作」（追加・削除）、右側は「成果を生む操作」（結合）。この左→右の流れはワークフローそのもの
- 結合ボタンだけが `.borderedProminent` で、画面全体の中で唯一の「ここを押せ」ポイント
- サイドバーとプレビューの比率は約 **1:2.5**。プレビューが主役であり、リストは脇役

- アイコンは大きく（48pt）、色は `.secondary`。主張しすぎず、でも確実に目に入る

### 6.6 アニメーション設計

> *"アニメーションは装飾ではない。因果関係を伝える言語だ。"*

| トリガー | アニメーション | Duration | Curve |
|---------|-------------|----------|-------|
| ファイル追加 | リストにスライドイン | 0.3s | `.spring(response: 0.3, dampingFraction: 0.8)` |
| ファイル削除 | リストからフェード+縮小して消滅 | 0.25s | `.easeOut` |
| 並び替え | 行が移動先へスムーズに移動 | 0.3s | `.spring(response: 0.3, dampingFraction: 0.7)` |
| ドラッグ進入 | ドロップゾーンが穏やかに発光 | 0.2s | `.easeInOut` |
| ドラッグ退出 | ドロップゾーンがフェードアウト | 0.15s | `.easeOut` |
| プレビュー切替 | クロスフェード | 0.2s | `.easeInOut` |
| 結合開始 | プログレスバーがスライドイン | 0.3s | `.easeOut` |
| 結合完了 | 成功バナーが上からスライドイン | 0.4s | `.spring(response: 0.4, dampingFraction: 0.75)` |
| 成功バナー消滅 | フェードアウト | 0.5s | `.easeOut`（5秒後に発火） |
| 空状態→リスト表示切替 | 空状態がフェードアウト→SplitViewが展開 | 0.35s | `.easeInOut` |
| モード切替 | 左右ペインのコンテンツがクロスフェード | 0.25s | `.easeInOut` |
| 分割境界線表示 | 区切り線がフェードイン | 0.2s | `.easeInOut` |
| 分割設定変更 | プレビューのグループ分けがアニメーション更新 | 0.3s | `.spring(response: 0.3, dampingFraction: 0.8)` |

**アニメーションの鉄則**:
- 0.5秒を超えるアニメーションは存在しない。ユーザーを待たせる演出は傲慢
- 全てのアニメーションに物理的な意味がある（上から落ちる、横からスライドする など）
- `withAnimation {}` で明示的にトリガーし、意図しない箇所が動かないようにする

## 7. ウィンドウ仕様

| 項目 | 値 | 理由 |

import Foundation

enum SplitMethod: String, CaseIterable, Identifiable {
    case pageRanges   // ページ範囲指定
    case everyPage    // 1ページずつ
    case oddEven      // 偶数/奇数

    var rangeText: String = ""

### Step 3: PDFSplitService の実装

**目的**: PDF分割のコアロジックを分離した純粋なサービスとして実装

**作成ファイル**: `Services/PDFSplitService.swift`

**実装内容**:
```swift
enum PDFSplitService {
    /// PDFファイルを分割し、指定フォルダに複数ファイルとして保存する
    /// - Parameters:
    ///   - url: 分割元PDFのURL
    ///   - groups: ページインデックスのグループ配列（0-based）
    ///   - outputDirectory: 保存先ディレクトリURL
    ///   - baseName: 出力ファイルのベース名
    ///   - progress: 進捗コールバック (0.0〜1.0)
    /// - Returns: 生成されたファイルのURL配列
    static func split(
        url: URL,
        groups: [[Int]],
        to outputDirectory: URL,
        baseName: String,
        progress: @Sendable (Double) -> Void
    ) async throws -> [URL]
}

enum PDFSplitError: LocalizedError {
    case fileNotReadable(URL)
    case passwordProtected(URL)
    case writeFailed(URL)
    case noGroups
}
```

**分割アルゴリズム**:
1. 元PDFを `PDFDocument(url:)` で読み込み（失敗→エラー）
2. `isLocked` チェック
3. 各グループごとにループ:
   a. 新規 `PDFDocument()` を生成
   b. グループ内のページインデックスを順に `page(at:)` → `insert(_:at:)`
   c. `{baseName}_part{n}.pdf` として `write(to:)` で保存
   d. 進捗を更新
4. 生成されたURL配列を返却

**`PDFMergeService` との共通点**: `PDFDocument` の読み込み・ページ挿入・書き出しパターンは結合サービスと同一。ただしサービスは分離を維持する（過度な共通化は避ける）

**完了条件**: ビルド成功。サービス単体でテスト可能な設計。

### Step 4: セキュリティスコープ管理の一元化

**目的**: `startAccessingSecurityScopedResource()` / `stopAccessingSecurityScopedResource()` の呼び出しを一箇所で管理し、リーク・未解放を防ぐ

**問題の本質**:

**作成ファイル**: `Services/SecurityScopeManager.swift`

```swift
import Foundation

/// セキュリティスコープ付きURLのライフサイクルを一元管理する
final class SecurityScopeManager {
    /// 現在アクセス中のURL集合
    private var accessingURLs: Set<URL> = []

    /// セキュリティスコープアクセスを開始する
    /// - Returns: アクセス成功なら true
    @discardableResult
    func startAccessing(_ url: URL) -> Bool {
        guard !accessingURLs.contains(url) else { return true }  // 二重開始防止
        guard url.startAccessingSecurityScopedResource() else { return false }
        accessingURLs.insert(url)
        return true
    }

    /// セキュリティスコープアクセスを終了する
    func stopAccessing(_ url: URL) {
        guard accessingURLs.contains(url) else { return }
        url.stopAccessingSecurityScopedResource()
        accessingURLs.remove(url)
    }

    /// 全てのアクセスを終了する（アプリ終了時）
    func stopAll() {
        for url in accessingURLs {
            url.stopAccessingSecurityScopedResource()
        }
        accessingURLs.removeAll()
    }

    deinit {
        stopAll()
    }
}
```

**変更内容**:
- `PDFMergeViewModel`（→ 後に `PDFWorkspaceViewModel`）に `let scopeManager = SecurityScopeManager()` を保持
- `ContentView` の `fileImporter` / `dropDestination` での `startAccessing` 呼び出しを `scopeManager.startAccessing` に統一
- `removeItems` / `clearAll` / `setSplitSource` での `stopAccessing` を `scopeManager.stopAccessing` に統一
- ドロップ時（結合モード）にも `scopeManager.startAccessing` を呼ぶよう修正

**完了条件**: ビルド成功。`startAccessingSecurityScopedResource` / `stopAccessingSecurityScopedResource` の直接呼び出しがアプリ内に存在しない（`SecurityScopeManager` 内を除く）。

### Step 10: ファイル間ページドラッグ&ドロップ

**目的**: 結合モードのページビューで、異なるファイルのセクション間でページを移動可能にする

**これは v1.3.0 最大の新機能であり、最も実装難易度が高い。**

**現状の制限**:
- `PageGridView` のページD&Dは**ファイル内のみ**（`ForEach` + `.onMove` のスコープ）
- ファイルをまたいだページ移動は不可

**設計方針**:

ページの「ファイル間移動」は、実質「ページのファイルAからの削除 + ファイルBへの挿入」：

```swift
/// ページをファイル間で移動する
func movePageBetweenFiles(
    pageID: PageItem.ID,
    fromFileID: PDFItem.ID,
    toFileID: PDFItem.ID,
    insertionIndex: Int
) {
    guard let srcIdx = merge.pdfItems.firstIndex(where: { $0.id == fromFileID }),
          let pageIdx = merge.pdfItems[srcIdx].pages.firstIndex(where: { $0.id == pageID }),
          let dstIdx = merge.pdfItems.firstIndex(where: { $0.id == toFileID })
    else { return }

    var page = merge.pdfItems[srcIdx].pages.remove(at: pageIdx)
    page = PageItem(
        parentID: toFileID,  // 親IDを更新
        pageIndex: page.pageIndex,
        thumbnail: page.thumbnail,
        isIncluded: page.isIncluded
    )
    merge.pdfItems[dstIdx].pages.insert(page, at: insertionIndex)
    dismissMergedPreview()
}
```

**UI実装**:
- `.draggable(page.id.uuidString)` は既に実装済み
- `.dropDestination(for: String.self)` を各ファイルセクションに追加
- ドロップ先のファイルセクションがハイライトされるフィードバック
- 同一ファイル内のドロップは既存の `.onMove` で処理

**`PageItem` の変更**:
- `parentID` を `var` に変更（移動時に更新が必要）
- `let id = UUID()` はそのまま維持（移動してもIDは不変）

**`PDFItem` の注意**:
- ページ移動後、`pageCount` プロパティは `pages.count` と乖離する
- `pageCount` を算出プロパティに変更するか、`originalPageCount` と `currentPageCount` を分離

```swift
struct PDFItem: Identifiable, Hashable {
    // ...
    let originalPageCount: Int   // 元ファイルのページ数（不変）
    // pageCount は pages.count で動的に算出
    var currentPageCount: Int { pages.count }
}
```

**完了条件**: ビルド成功。ページビューでファイルAのページをファイルBのセクションにドラッグ&ドロップでき、ページが移動する。移動後のページが結合結果に正しく反映される。
